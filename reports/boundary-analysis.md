# Boundary analysis

**Scope:** current repository worktree, reviewed 2026-07-11. This is a path-and-contract analysis: it identifies where the documented purpose of a location overlaps with another location or leaves ownership unclear. It does not claim that the underlying research assertions are incorrect, and it proposes no file moves, renames, or migrations.

## Boundary rule used

A repository object needs one primary owner:

- **Knowledge** owns a public, reusable fact about a real-world entity.
- **Analysis** owns a bounded interpretation, comparison, shortlist, or calculation based on knowledge.
- **Workspace** owns an applicant's private or operational decision process; it never becomes public evidence by itself.
- **Generated** owns a reproducible projection of other owned material; it owns neither the source fact nor the interpretation.

Links, citations, and deliberately empty future namespaces are not duplication. Repeated prose, competing claims of canonical ownership, or two locations that invite the same new record are boundary risks.

## Major-directory classification

Each major top-level directory is assigned **exactly one** category below. The final column records a secondary pressure without assigning a second category.

| Directory | Category | Primary responsibility | Secondary pressure / boundary note |
| --- | --- | --- | --- |
| `.idea/` | Configuration | Local IDE state. | It is ignored by `.gitignore`; it should remain outside shared project knowledge. |
| `advisor-due-diligence/` | Analysis | Applicant-specific evidence review, comparison, decision preparation, and outreach hypotheses. | It is specific to a declared applicant, but it is not the interaction-record workspace; dated private interaction facts remain in `relationships/`. |
| `assets/` | Platform | Shared non-content assets used by repository presentation or tooling. | Only tracked `.gitkeep` files exist now, so its allowed asset types are not yet documented. |
| `countries/` | Knowledge | Legacy, geography-oriented research records and country context. | It also contains shortlists and decisions, creating the largest current knowledge/analysis mix. |
| `docs/` | Documentation | Human-readable governance, architecture, policy, ADR, and release guidance. | Some pages call themselves contracts, which can compete with schemas and versioned scoring files for authority. |
| `ecosystems/` | Knowledge | Ecosystem-level records and context. | Its intended records overlap with the vNext `entities/ecosystems/` home and ecosystem reports. |
| `entities/` | Knowledge | Canonical vNext entity records, organized by entity type. | It is intentionally an empty target namespace; it must not acquire report or workspace material. |
| `framework/` | Platform | Entity-record starter templates and reusable structural scaffolding. | Its template role is easy to confuse with `templates/`, which serves analyses instead. |
| `methodology/` | Documentation | Metric definitions and evidence-collection guidance. | It is related to scoring but should not own versioned score outputs or applicant decisions. |
| `principal-investigators/` | Knowledge | PI discovery/index navigation. | Its README calls it a canonical PI index while vNext documentation assigns canonical records to `entities/principal-investigators/`. |
| `relationships/` | Workspace | Research Relationship Management (RRM): deliberately recorded interaction process. | Its name can be confused with the graph relationships described in `docs/data-model/` and `docs/architecture/`. |
| `reports/` | Analysis | Published, evidence-bounded comparisons, shortlists, and other interpretations. | It also currently contains a source register, a personal roadmap, and redirect-style requested-list files. |
| `research-groups/` | Knowledge | Research-group discovery and future group records. | It competes prospectively with `entities/research-groups/` and report-based group queues. |
| `research-leaders/` | Analysis | Deep, evidence-backed leader dossiers and candidate-specific interpretation. | Individual dossier files contain factual material, but the package includes career-value and contact-strategy conclusions, so it is not the vNext canonical entity store. |
| `schemas/` | Platform | Machine-checkable structural contracts for metadata. | It must remain distinct from explanatory frontmatter and architecture documentation. |
| `scoring/` | Platform | Versioned scoring-model and weighting contracts. | A profile-specific scorecard currently shares `v1/` with general model contracts and is therefore an analysis artifact in a platform namespace. |
| `scripts/` | Automation | Reproducible validation, ingestion, and generation tooling. | It is presently a reserved placeholder, so no output should be described as generated yet. |
| `templates/` | Platform | Report and comparison starter templates. | Its scope overlaps in name, not intent, with `framework/`. |
| `views/` | Generated | Derived navigation over canonical entities. | It currently contains architecture-only definitions, not generated indexes; that target status must stay explicit. |

There is no top-level **Archive** directory. `relationships/archived/` is correctly a workspace lifecycle state rather than a repository-wide archive. No additional category should be invented merely to classify it.

## What is already separated well

These are useful boundaries to preserve rather than “fix”:

- The current RRM records explicitly point to due-diligence material instead of duplicating it; see [`relationships/active/adem-tekin/meeting-preparation.md`](../relationships/active/adem-tekin/meeting-preparation.md).
- The requested-list report files are short navigation aliases to [`special-reports.md`](special-reports.md), not independent copies of the lists.
- The v1/vNext documentation explicitly says that no records are being moved yet, for example in [`docs/architecture/entity-model.md`](../docs/architecture/entity-model.md) and [`entities/README.md`](../entities/README.md). The issue is not the staged design; it is the need for a consistently visible entry rule while both models coexist.

## Recommended boundary decisions before any migration

1. Adopt the one-owner rule and the directory classification above as a temporary operating policy.
2. Add a compact record-location decision tree to the repository navigation: canonical fact, analysis, personal process, or derived view.
3. Resolve the three competing uses of “canonical” for PI, group, and ecosystem material in prose before creating any new record.
4. Publish authority ladders for metadata and scoring so that a schema, metric definition, or score formula has one named source of truth.
5. Make profile-specific visibility explicit before additional personal career, contact, or applied-score artifacts are added outside the workspace boundary.

None of these recommendations requires moving, renaming, deleting, or rewriting existing knowledge. They establish the decision boundaries needed to make a later migration safe and reviewable.

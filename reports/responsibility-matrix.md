# Responsibility matrix

> **Scope:** current working tree, reviewed 2026-07-11. “Owner” means the logical steward of the content boundary, not an assigned individual. The matrix states the intended responsibility of each current top-level location without moving or changing any content.

## Top-level files

| File | Owner | Inputs → outputs | Allowed contents | Forbidden contents | Dependencies / duplication risk |
| --- | --- | --- | --- | --- | --- |
| `.DS_Store` | Local workstation | Finder state → Finder metadata | None as repository content | Any committed project artifact | No dependency; it is an out-of-bound workspace file. |
| `.gitignore` | Repository operations | Ignore policy → Git exclusion behavior | Stable ignore rules | Project knowledge or per-user secrets | Governs `.idea`; currently does not cover the present `.DS_Store`. |
| `README.md` | Documentation steward | Architecture/status summaries → entry-point navigation | Concise repository orientation and links | Detailed duplicate contracts or report bodies | Depends on all public top-level layers; highest navigation-duplication risk. |
| `ROADMAP.md` | Architecture steward | Approved direction → phased plan | Future phase boundaries and sequencing | Task logs or migration execution details | Depends on architecture and release status. |
| `CHANGELOG.md` | Release steward | Released changes → history | Dated release notes | Full design rationale | Depends on release notes/commits; avoid duplicating roadmap. |
| `CONTRIBUTING.md` | Community steward | Evidence and workflow rules → contributor instructions | Contribution process | Duplicate full methodology/schema specs | Depends on docs and templates. |
| `CODE_OF_CONDUCT.md` | Community steward | Community policy → conduct expectations | Conduct and enforcement policy | Research analysis | Independent governance document. |
| `SECURITY.md` | Security steward | Vulnerability-reporting policy → reporting route | Security disclosure guidance | Credentials, vulnerabilities, research content | Independent governance document. |
| `CITATION.cff` | Citation steward | Citation metadata → machine-readable citation | CFF fields | Narrative documentation | Referenced by README; no duplicate citation record. |
| `LICENSE` | Legal steward | License terms → reuse permission | License text | Project methodology or personal clauses | Referenced by README; authoritative legal text. |

## Operational rules derived from the matrix

1. A public fact is authored once in the applicable canonical knowledge record; reports, views, and workspaces link to it.
2. A conclusion for a declared applicant belongs in analysis, never in the fact owner.
3. An interaction or next action belongs in the workspace layer and never upgrades public evidence by itself.
4. Reusable schemas, scoring rules, templates, and scripts are platform/configuration material; they do not own research findings.
5. A generated view can be regenerated from its declared inputs. If it cannot, it is an analysis or a manually curated knowledge index and should say so.
